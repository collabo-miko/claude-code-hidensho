### 5. 不要ファイル読み込みの制限（deny ルール体験）

ビルド成果物や依存パッケージの読み込みを `deny` ルールでブロックする効果を体験します。

**Step 1: デモ用ビルド成果物を生成**


以下のダミーファイルが生成されます:

| ディレクトリ | 内容 | 実プロジェクトでのサイズ目安 |
|-------------|------|--------------------------|
| `node_modules/` | lodash, express のダミー | 数百MB |
| `dist/` | ミニファイ済みバンドル + ソースマップ | 数MB |
| `build/` | ビルドログ | 数KB〜数MB |
| `coverage/` | テストカバレッジレポート | 数MB |

**Step 2: deny ルールの効果を確認**

```bash
#   node_modules/, dist/, build/, coverage/ は deny でブロックされる
```

**Step 3: deny ルールを一時的に無効化して比較**

```bash
# Claude のセッション内で:
# 「.claude/settings.json の deny ルールを全てコメントアウトして」
# → もう一度「全ファイルを調べて」と依頼

# → /context でコンテキスト消費の差を確認
#
# 確認後、deny ルールを元に戻すのを忘れずに！
```

**Step 4: deny ルールの設定を確認**

`.claude/settings.json` の `deny` セクションを確認してください:

```json
{
  "permissions": {
    "deny": [
      "Read(node_modules/**)",  // 依存パッケージ（最大の節約効果）
      "Read(dist/**)",          // ビルド出力
      "Read(build/**)",         // ビルド成果物
      "Read(.next/**)",         // Next.js キャッシュ
      "Read(coverage/**)",      // テストカバレッジ
      "Read(*.lock)",           // ロックファイル（数千〜数万行）
      "Read(*.min.js)"          // ミニファイ済み JS
    ]
  }
}
```

**Step 5: クリーンアップ**

```bash

bash cleanup-demo.sh
```

> **ポイント:** deny ルールはプロジェクトのビルド構成に合わせてカスタマイズしましょう。
> Python なら `Read(.venv/**)`, `Read(__pycache__/**)` なども有効です。

## Hooks によるコスト最適化（実践編）

Hooks を使うと、プロンプトに頼らず**確実に**トークン消費を抑制できます。
このサンプルには 2 つのフックスクリプトと PostCompact フックが設定済みです。

### 8. テスト出力フィルタリング（filter-test-output.sh）

テストコマンドの出力を「失敗箇所のみ」に自動フィルタリングします。

```bash
# まずスクリプトの中身を確認
# 「filter-test-output.sh を読んで仕組みを説明して」

# 動作確認: テストを実行してフィルタの効果を観察
npx jest --verbose
# → フックが出力を FAIL/ERROR 行 + 前後の文脈のみに絞る
# → 1000行の出力が 50〜100行に圧縮される
```

**仕組み:**
- `PreToolUse`（Bash マッチャー）でテストコマンドを検出
- `hookSpecificOutput` の `updatedInput` でコマンドを書き換え
- `grep -E -A 5 -B 2 'FAIL|ERROR'` で失敗箇所のみ抽出
- `head -100` で最大行数を制限

**試してみよう:**
1. `.claude/settings.json` の `filter-test-output.sh` フックを一時的にコメントアウト
2. 同じテストを実行して出力量の違いを比較
3. `/context` でコンテキスト消費の差を確認

### 9. コマンド出力のトランケート（truncate-output.sh）

`find` や `ls -R` など大量出力が予想されるコマンドを自動で 200 行に制限します。

```bash
# 動作確認: 大量出力が予想されるコマンドを実行
find . -name "*.js"
# → フックが出力を 200行に制限

# head/tail が既にパイプされている場合はスキップされる
find . -name "*.js" | head -5
# → フックは介入しない（ユーザーが意図的に制限済み）
```

**仕組み:**
- `find`, `ls -R`, `git log --all` などのパターンを検出
- 既に `head`/`tail` がパイプされていれば介入しない
- `hookSpecificOutput` でコマンドに `| head -n 200` を追加

**カスタマイズ:**
```bash
# truncate-output.sh の MAX_LINES を変更して制限行数を調整
MAX_LINES=200  # デフォルト値。プロジェクトに合わせて調整可能
```

```bash
# このサンプルでは CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=75 で

# 3. /context でコンテキスト使用量を確認する
```

### 11. ビルド出力フィルタリング（filter-build-output.sh）

ビルドコマンドの大量ログを WARNING/ERROR のみに自動圧縮します。

```bash
# スクリプトの中身を確認
# 「filter-build-output.sh を読んで仕組みを説明して」

# 動作テスト（npm run build がある場合）
npm run build
# → 成功時: 最後の3行（サマリー）のみ表示
# → 失敗時: WARNING/ERROR 行 + 前後の文脈のみ抽出（最大80行）
# → 全出力は /tmp/_build_full.log に保存される
```

**仕組み:**
- `tee` でフル出力をファイルに保存しつつ、パイプで `grep` フィルタに通す
- `PIPESTATUS` でビルド自体の exit code を保持（フィルタの成否ではなくビルドの成否を返す）
- 成功時は `tail -3` でサマリーのみ、失敗時はエラー行に集中

**テスト出力フィルタとの違い:**

| フック | 対象 | フィルタ方式 | 全出力の保存 |
|--------|------|-------------|-------------|
| filter-test-output.sh | jest, pytest 等 | FAIL/ERROR 行を抽出 | なし |
| filter-build-output.sh | npm build, tsc 等 | WARNING/ERROR 行を抽出 | `/tmp/_build_full.log` に保存 |

### 12. サブエージェントに冗長な作業を委譲

テストスイートの実行、大量ログの分析など**出力が多い作業**をサブエージェントに委譲すると、
メインのコンテキストには要約のみが返り、トークンを大幅に節約できます。

**Step 1: サブエージェントなしでテスト実行**

```bash
# Claude のセッション内で:
# 「src/ 配下のコードを読んで、テストを書いて実行して」
# → メインのコンテキストにテストコード + 実行結果が全て入る
# → /context でコンテキスト消費を確認
```

**Step 2: サブエージェントに委譲**

```bash
# Claude のセッション内で:
# 「src/ 配下のコードのテストを書いて実行して。
#   テストの作成と実行はサブエージェントに任せて、結果の要約だけ教えて」
#
# → サブエージェントがテスト作成・実行を担当
# → メインには「3テスト中3テスト合格」のような要約のみ返る
# → /context でコンテキスト消費を比較
```

**Step 3: サブエージェントのモデルを下げる**

```bash
# Claude のセッション内で:
# 「haiku のサブエージェントで src/ のテストを実行して、結果を要約して」
#
# → サブエージェントは haiku で動作（さらにコスト削減）
# → メインは opus/sonnet のまま高品質な判断ができる
```

**効果の目安:**

| 方式 | コンテキスト消費 | コスト |
|------|----------------|--------|
| メインで直接テスト実行 | テストコード + 全出力（数千〜数万トークン） | 高 |
| サブエージェントに委譲 | 要約のみ（500〜1,000トークン） | 低 |
| haiku サブエージェントに委譲 | 要約のみ + haiku 単価 | 最低 |

> **ポイント:** サブエージェントへの委譲が特に有効なタスク:
> - テストスイートの実行と結果分析
> - 大量のログファイルから特定パターンの検索
> - ドキュメントやコードの一括フォーマット
> - 依存関係の脆弱性スキャン結果の解析

## この設定に含まれるコスト最適化

| 設定 | 効果 |
|------|------|
| deny: Read(node_modules/**) 等 | 不要ファイル読み込み防止 |
| CLAUDE_AUTOCOMPACT_PCT_OVERRIDE: 75 | 早めの自動コンパクション |
| MAX_THINKING_TOKENS: 8000 | Extended Thinking の予算制限 |
| ENABLE_TOOL_SEARCH: auto:5 | MCP ツール遅延読み込み |
| **filter-test-output.sh** | **テスト出力を失敗箇所のみに圧縮** |
| **filter-build-output.sh** | **ビルド出力を WARNING/ERROR のみに圧縮** |
| **truncate-output.sh** | **大量出力コマンドを200行に制限** |
| **PostCompact フック** | **コンパクション発生を通知** |

## ポイント

- `/context` を定期的に確認する習慣をつける
- 単純な作業に `opus` + `high` を使わない
- 曖昧なプロンプトは20以上のファイルを読み込むが、具体的なプロンプトは2〜3ファイルで済む
- CLAUDE.md の詳細な手順はスキルに分離してオンデマンドロード
- **Hooks はプロンプトと違い「必ず毎回」実行される — コスト最適化の確実な防衛線になる**
- **フックの効果は `/context` でビフォー・アフターを比較して定量的に検証する**
