## 実践編: ダメな CLAUDE.md のリファクタリング

### 5. アンチパターンの分析

`CLAUDE.md.bad-example` は意図的に7つのアンチパターンを詰め込んだ悪い例です。

```bash
# セッション内で:
# 「CLAUDE.md.bad-example を読んで、問題点を分析して」
```

**含まれるアンチパターン:**

| # | 問題 | 影響 |
|---|------|------|
| 1 | 否定形の多用（「〜するな」× 20個） | 遵守率が下がる |
| 2 | 同じ内容の重複記述（5回） | 無駄なトークン消費 |
| 3 | 曖昧な指示（「きれいなコードを書く」） | 効果なし |
| 4 | 過度に詳細な手順（PR/デプロイ手順） | スキルに分離すべき |
| 5 | 古い情報・矛盾する記述 | 誤った出力の原因 |
| 6 | 無関係な情報（MTG 予定等） | トークンの無駄 |
| 7 | 全体で 200行超え | 毎セッションのコスト増 |

### 6. リファクタリング演習

```bash
# セッション内で:
# 「CLAUDE.md.bad-example を元に、200行以内の良い CLAUDE.md を作って。
#   以下のルールに従って:
#   - 否定形を肯定形に書き換える
#   - 重複を排除する
#   - 曖昧な指示を具体化する
#   - PR/デプロイ手順はスキルに分離する指示だけ残す
#   - 古い情報を更新する
#   - 無関係な情報を削除する」
```

**リファクタリング前後の比較例:**

```
❌ 悪い例: var を使うな / any を使うな / console.log を使うな
✅ 良い例: const/let を使用する（const 優先）。型は明示的に宣言する。
           ログは構造化ロガー（winston/pino）を使用する。

❌ 悪い例: きれいなコードを書く
✅ 良い例: 関数は20行以内。ネストは3段まで。

❌ 悪い例:（PR レビュー手順を 15行で記述）
✅ 良い例: PR レビュー手順は `/review-pr` スキルを参照
```

### 7. ルール遵守率の検証

リファクタリング後の CLAUDE.md が実際に機能するか検証します。

```bash
# Step 1: リファクタリング後の CLAUDE.md を適用
# Step 2: テストプロンプトでルール遵守を確認

# 「src/utils/helpers.js に formatCurrency 関数を追加して」
# → 確認項目:
#   - ES modules で書かれているか（CommonJS でないか）
#   - const が使われているか
#   - JSDoc コメントが付いているか
#   - テストも一緒に生成されたか

# 「src/api/users.js に DELETE エンドポイントを追加して」
# → 確認項目:
#   - RFC 7807 のエラー形式が使われているか（パス限定ルール）
#   - ステータスコードが適切か（204）
#   - 入力バリデーションがあるか
```

## ポイント

- CLAUDE.md は **200行以内** に収める（短いほど遵守率が高い）
- 肯定形で書く（「〜しないこと」より「〜すること」）
- 最重要ルールは冒頭と末尾に配置
- `@path/to/file` でファイル内容を参照させられる
- **手順やプレイブックはスキルに分離してオンデマンドロード**
- **リファクタリング後は必ずテストプロンプトで遵守率を検証する**
